> 把 s05 的内存 SessionManager 接到磁盘上：四个文件、append-only 的 `history.jsonl` + atomic-replace 的 `MEMORY.md` / `SOUL.md` / `USER.md`，再加一个独立的 `.cursor` 让 s10 的 Consolidator 知道"读到哪一行了"。
> Wire s05's in-memory SessionManager to disk: four files — append-only `history.jsonl` plus atomic-replace `MEMORY.md` / `SOUL.md` / `USER.md`, with a separate `.cursor` file so s10's Consolidator can track "how much of the log have I summarized".

### File schema / 文件结构

```
<workspace>/
├── history.jsonl   # one HistoryEntry per line, append-only
├── .cursor         # last successfully appended cursor (atomic replace)
├── MEMORY.md       # long-term facts, atomic replace (LLM-curated; s10/App.A)
├── SOUL.md         # agent persona, atomic replace
└── USER.md         # user preferences, atomic replace
```

`AGENTS.md` / `TOOLS.md` are part of `BootstrapFiles` (s08 reads them) but
are typically authored by humans / generated by the runtime — MemoryStore
provides the read path, not a writer.

### Test / 测试

```bash
go test -count=1 -race ./...
```

The concurrency test fires 10 goroutines, each appending one HistoryEntry;
final line count must be 10 and every line must parse as JSON. Race detector
catches any leak in the `appendMu` mutex.

### Teaching points / 教学要点

1. **Append-only for history; atomic replace for curated memory.** `history.jsonl` is the audit trail — never lose a turn. `MEMORY.md` is the LLM's interpretation — it can be revised (Dream cycle in App. A). Two write semantics for two different roles.
2. **`os.Rename` is the atomicity primitive on POSIX.** Write the new contents to a temp file in the same directory, then rename onto the target. POSIX guarantees rename within a filesystem is atomic — a crash leaves either old or new, never half-and-half.
3. **`.cursor` is a separate file.** It could be derived from `history.jsonl` (count the lines) but isn't, because the cursor advances only when *consolidation* succeeds. Keeping it separate decouples "how much exists" from "how much is summarized" — exactly the seam s10 will use.
4. **`O_APPEND` + mutex for concurrent appends.** POSIX guarantees `O_APPEND` writes ≤ PIPE_BUF (typically 4 KB) are atomic, but our JSON lines can exceed that. The internal `sync.Mutex` covers the gap.
